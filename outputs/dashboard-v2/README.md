# Приемная кампания 360: dashboard v2

Первая версия сделана как самодостаточный web-прототип утвержденного дашборда. Данные интерфейса лежат в `src/data.js`; этот слой позже можно заменить внутренним API университета.

## Что показано в текущем примере

- Агрегированный срез по направлению `13.03.01 Теплоэнергетика и теплотехника` из файла `130301.xlsx`: 82 заявления, 65 заявлений на федеральный бюджет, 17 на платное, 4 согласия, медианный балл 211, 8 абитуриентов с баллом 240+.
- План приема из приложенных материалов: 25 бюджетных и 55 платных мест; стоимость очной формы — 427 000 рублей в год.
- Динамика заявлений рассчитана по датам появления личных дел в выгрузке. Сценарий оборота и потенциальный интерес являются демонстрационными.
- Персональные данные из исходной таблицы в интерфейс не выводятся: используются только агрегаты.
- Статус Bitrix24 и база 100 000 контактов показаны как частично подтвержденный источник, для которого еще требуется карта полей и правила сопоставления.

## Как открыть

Откройте `index.html` в браузере или запустите папку на любом локальном web-сервере.

Интерфейс адаптирован для экранов от 320 px: на телефоне боковая навигация становится горизонтальной, показатели перестраиваются в одну или две колонки, а широкая таблица прокручивается пальцем по горизонтали. Это тот же адрес сайта — отдельная мобильная ссылка не нужна.

## Как развивать до онлайн-версии

1. Внутренний сервер университета принимает данные из приемной комиссии, CRM Bitrix, мероприятий и рекламных кабинетов.
2. Данные нормализуются по уникальному коду абитуриента и хранятся внутри периметра университета.
3. Локальная нейросеть получает только внутренние данные, формирует сегменты, риск недобора, следующее лучшее действие и персональные тексты.
4. Дашборд читает агрегаты и списки задач через закрытый API.
5. Bitrix получает готовые персональные сообщения и задачи для менеджеров без передачи чувствительных данных наружу.

## Минимальная архитектура

- Frontend: статический web-интерфейс, который можно развернуть на внутреннем сервере.
- Backend API: FastAPI или Node.js внутри инфраструктуры университета.
- Storage: PostgreSQL для абитуриентов, касаний, заявлений, согласий и коммуникаций.
- AI layer: локальная LLM + правила скоринга, без облачных отправок персональных данных.
- Audit: журнал действий, версий рекомендаций и отправленных сообщений.
